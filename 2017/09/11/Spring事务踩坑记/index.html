发表于 2017-09-11

使用spring 事务踩到的坑

这段期间使用使用 spring 事务遇到几个深坑，遂记录下来。

坑1:@Transactional 作用非 public 方法

这个问题使用 spring 的过程中中一直没发现，直到有一天升级 idea 以后发现 idea 给我如下错误：

于是按照 idea 的提示修改，将方法设置为 public。没有深究其中的原因，本以为这样修改以后就不会有其他问题。于是，踩到第二个坑。

坑2：类内部调用 @Transactional 方法

一次项目中，一个类内部调用带有 @Transactional 的方法，目的是为了方法内部发生异常时，回滚数据。项目上线后，方法内部果然发生异常，日志也显示了方法抛出的异常，但是数据的状态却未回滚。

碰到这两个问题，我才意识到原来对 spring 还是这么不熟悉。于是查看 spring 事务方面的文档，带着疑问，阅读 spring 的文档。

spring 的事务调用过程

文档用一张图片介绍事务方法调用过程。

从图中可以看到，调用者调用的是 AOP 的代理对象，事务通知在最外层，其他通知被包围。

@Transactional 只能作用有 public 方法

Method visibility and @Transactional

When using proxies, you should apply the @Transactional annotation only to methods with public visibility. If you do annotate protected, private or package-visible methods with the @Transactional annotation, no error is raised, but the annotated method does not exhibit the configured transactional settings. Consider the use of AspectJ (see below) if you need to annotate non-public methods

当使用代理，你只能将 @Transactional 作用于 public 方法上。如果作用于其他，事务将不起到作用。

外部方法调用 @Transactional 才有效

In proxy mode (which is the default), only external method calls coming in through the proxy are intercepted. This means that self-invocation, in effect, a method within the target object calling another method of the target object, will not lead to an actual transaction at runtime even if the invoked method is marked with @Transactional. Also, the proxy must be fully initialized to provide the expected behaviour so you should not rely on this feature in your initialization code, i.e. @PostConstruct

默认的代理模式，外部方法调用才会有效。

看过以上文档，可以知道使用 spring 的事务，要避免以上的情况。那么为什么上述情况不能事务不起作用那。我们可以看到，会看上面 spring 事务发生的过程，其里面本质使用了 AOP 。而 spring 的 AOP构建与动态代理的基础上，因此 Spring 对AOP 的支持局限于方法拦截。

Spring AOP currently supports only method execution join points (advising the execution of methods on Spring beans). Field interception is not implemented, although support for field interception could be added without breaking the core Spring AOP APIs. If you need to advise field access and update join points, consider a language such as AspectJ.
